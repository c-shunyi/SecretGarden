# 文件功能技术方案（MVP）

## 1. 目标
- 文件必须先上传到后端，再由后端保存文件并写入数据库元数据。
- 业务模块在需要展示图片/文件的位置，统一关联文件表数据，不在业务表里直接存物理路径。
- 日记图片关系表正式使用 `diary_files`，不再使用 `diary_images`。

## 2. 适用范围
- 用户头像
- 日记图片（最多 9 张）
- 后续可扩展到纪念日封面、评论图片等

## 3. 设计原则
- 单一文件中心：所有上传文件统一进入 `files` 表管理。
- 业务解耦：业务表只保存文件关联关系，不直接耦合存储实现。
- 安全优先：后端校验类型、大小、权限，禁止前端直传业务目录。

## 4. 上传流程
1. 前端通过 `multipart/form-data` 请求上传文件到后端接口。
2. 后端做文件校验（MIME、扩展名、大小、用户身份）。
3. 后端保存文件到服务器目录（MVP）或对象存储（后续）。
4. 后端写入 `files` 表，返回 `file_id` 和可访问地址。
5. 前端在业务提交时传入 `file_id`，后端再写入业务关联表。

## 5. 数据库设计

### 5.1 文件主表：`files`
建议字段：
- `id` bigint PK
- `biz_type` varchar(32) 文件业务类型（`avatar`/`diary_image`/`anniversary_cover`）
- `storage` varchar(16) 存储类型（`local`/`oss`）
- `bucket` varchar(128) 可空（对象存储桶）
- `object_key` varchar(255) 存储键或相对路径
- `origin_name` varchar(255) 原始文件名
- `ext` varchar(20) 扩展名
- `mime_type` varchar(100)
- `size` bigint 字节大小
- `sha256` char(64) 可选，去重/校验
- `width` int 可空（图片）
- `height` int 可空（图片）
- `uploader_id` bigint 上传人
- `status` tinyint 默认 1（1=有效，0=删除）
- `created_at` datetime
- `updated_at` datetime
- `deleted_at` datetime 可空

索引建议：
- `idx_files_uploader_id`
- `idx_files_biz_type`
- `idx_files_sha256`

### 5.2 业务关联方式
推荐“业务独立关联表”，便于约束和扩展：

1. 用户头像（单文件）
- 在 `users.avatar_file_id` 存 `files.id`（一对一）

2. 日记图片（一个 diary 对应多条 diary_files，1:N）
- 使用 `diary_files`（正式替代 `diary_images`）
  - `id`
  - `diary_id`
  - `file_id`
  - `sort_order`
  - `created_at`
  - 约束：`UNIQUE(diary_id, file_id)`，避免重复关联同一文件
  - 约束：`UNIQUE(diary_id, sort_order)`，保证排序唯一
  - 规则：单条日记最多 9 条关联记录（服务端校验）

3. 纪念日封面（可选）
- 在 `anniversaries.cover_file_id` 存 `files.id`

## 6. 接口约定（建议）

### 6.1 上传文件
- `POST /api/v1/files/upload`
- 请求：`multipart/form-data`
  - `file`: 文件二进制
  - `bizType`: 业务类型（如 `avatar`、`diary_image`）
- 响应：
```json
{
  "id": 1001,
  "url": "https://api.example.com/uploads/2026/02/25/xxx.jpg",
  "bizType": "diary_image",
  "size": 123456,
  "mimeType": "image/jpeg"
}
```

### 6.2 删除文件（软删）
- `DELETE /api/v1/files/:id`
- 权限：仅上传者本人或管理员

### 6.3 业务提交示例（日记）
- `POST /api/v1/diaries`
- 请求体：
```json
{
  "content": "今天一起散步了",
  "fileIds": [1001, 1002]
}
```
- 服务端写入：
  - `diaries`
  - `diary_files`

```text
  routes/
    file.js
uploads/
```

## 8. 安全与限制
- 仅允许白名单类型：`jpg/jpeg/png/webp/gif`
- 单文件大小限制：建议 `<= 10MB`
- 图片数量限制：日记最多 `9` 张
- 文件名不可信，服务端重命名并生成唯一 `object_key`
- 返回公开 URL 前进行鉴权策略确认（私有资源可改签名 URL）

## 9. 生命周期与清理
- 业务解绑策略：
  - 删除业务记录时只删除关联，不立即删物理文件
  - 定时任务清理“未被引用且超过保留期”的文件
- 建议增加引用计数或反查关联关系，避免误删

## 10. 实施顺序（建议）
1. 建 `files`、`diary_files` 表并完成迁移。
2. 实现 `POST /files/upload` 和本地存储。
3. 接入头像、日记发布，改为使用 `file_id` 关联。
4. 增加文件删除、垃圾文件清理任务。
5. 预留从本地存储迁移到对象存储的适配层。
